# NetDiag 对话驱动诊断流程（工程师模式）

## 目标
把 AI 当成网络工程师：用户用自然语言描述现象，AI 负责追问、下发只读命令、基于证据逐轮收敛根因。

## 交互主流程
1. 用户在聊天框输入：问题现象 + 设备 IP + 时间描述（可相对时间）。
2. 系统做意图解析：
   - 优先 AI 解析
   - 失败自动回退本地规则
   - 自动归一全角 IP（如 `192。168。0。88`）。
3. 缺关键信息时，AI 用工程师语气追问；若用户要求“AI 自行判断时间”，使用默认近 2 小时并显式声明假设。
4. 会话创建后自动进入诊断循环：
   - Baseline（每会话只采集一次）
   - Plan（给出本轮只读命令）
   - Approve
   - Execute
   - Analyze
   - Next Round / Conclude
5. 执行层去重：同一会话跨轮相同 `device_id + command` 复用历史结果，避免重复打设备。

## 自动批准策略
- 自动批准菜单按步骤开关：`create/baseline/plan/approve/execute/analyze/next_round/conclude`。
- 当下一步被勾选，UI 在聊天中给出“将自动继续执行”的轻提示，并自动推进，不再要求重复点击。

## 页面布局（白色 VSCode 风格）
- 左侧：页签导航（固定）。
- 中间：AI 对话（简洁，只保留用户消息、AI状态、动作卡、命令/执行折叠卡）。
- 右侧上半：分析显示区（实时显示 AI 推理、计划、证据、结果 JSON）。
- 右侧下半：设备终端回显（实时尾部输出）。

## 实时可观测性
- AI 每次动作状态、命令清单、执行摘要、分析结论都会同步到“分析显示区”。
- 点击聊天中的折叠卡可将详细内容投送到右侧分析区。
- 设备命令回显独立显示在右下终端区，便于对照 AI 结论与设备证据。

## 故障定位收敛标准
- 证据链必须包含：设备、时间、命令、输出片段。
- 结论必须可被只读命令证据复核。
- 推荐收敛（conclude）仅在 stop_decision 明确建议后触发。

# NetDiag 想象版需求文档（Vision）

## 1. 目标形态
把 NetDiag 从“诊断工具”演进为“网络故障数字副本平台”：
- 自动回放历史状态与配置变化
- 自动生成可执行 SOP 草案
- 自动评估风险与变更影响范围
- 始终坚持人审批准执行

## 2. 想象版能力清单
### 2.1 设备数字画像（Digital Twin Lite）
- 每台设备维护长期画像：
  - 拓扑角色、关键邻居、关键链路、常态资源曲线
  - 常见故障模式与恢复时间
- 新故障发生时先做“偏离检测”再规划命令。

### 2.2 跨系统证据总线
- 统一接入：
  - 设备 CLI 证据
  - Zabbix 指标
  - Syslog/事件流
  - 变更单系统（CAB/ITSM）
- AI 不直接读原始系统，而是读标准化证据事件（Evidence Bus）。

### 2.3 自适应 SOP 编排器
- 按“问题类型 + 厂商 + 历史成功案例”生成 SOP 草案
- 每一步带：
  - 目标
  - 预期信号
  - 失败分支（Plan B）
  - 停止条件

### 2.4 配置与变更因果引擎
- 自动把故障窗附近变更映射到疑似根因：
  - “哪条配置变化可能导致这组症状”
  - “若回滚，风险影响范围是什么”
- 输出变更因果解释图。

### 2.5 组织级知识飞轮
- 单次会话 -> 案例 -> SOP 模板 -> 命令库权重更新 -> 下一次更快定位
- 支持团队协作批注、复盘评分、专家审阅流程。

## 3. 产品体验（想象）
- 首页直接输入“故障时间+现象”，系统自动拉取：
  - 设备状态基线
  - 历史配置差异
  - Zabbix 指标变化
  - 已知问题和历史案例
- 返回一页式“诊断驾驶舱”：
  - 根因候选（带置信区间）
  - 证据链时间轴
  - 建议动作清单（需审批）
  - 预计恢复路径与验证项

## 4. 关键约束
- 不做无审批自动配置变更（默认永不自动下发）
- 任何结论必须可追溯到证据
- 多厂商命令策略必须可审计

## 5. 分阶段演进路线（想象）
- Phase A：状态库+配置库+案例库闭环
- Phase B：跨系统证据总线与因果分析
- Phase C：数字画像与自适应 SOP
- Phase D：组织级运维知识网络（跨团队共享）

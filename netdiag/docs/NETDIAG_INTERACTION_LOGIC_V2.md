# NetDiag AI 交互逻辑 V2（工程师式严格流程）

## 1. 目标
- 仅围绕网络设备故障诊断。
- 只执行只读命令：`show` / `display` / `dis`。
- 让 AI 按“证据驱动、逐轮收敛”的方式推进，不乱序、不回退、不重复采集。

## 2. 最小输入（对话中自然语言）
- 问题描述（question）
- 设备 IP（device_ip）
- 故障时间窗（fault_start / fault_end）

系统先做两类守卫：
1. 连接守卫：连接控制页签里的 username/password/timezone/timeout 是否完整。
2. 时间守卫：用户时间窗与设备时钟校准是否完成（Baseline）。

## 3. 严格状态机（后端强约束）
会话下一步动作由后端统一计算（`next_action`），前端只展示与触发，不自行越权：

1. `baseline`（仅一次）
2. `plan`
3. `approve`
4. `execute`
5. `analyze`
6. `next_round` 或 `conclude`

如果当前步骤正在运行，统一返回 `wait/busy`，不会重复触发同一步。

## 4. 轮次闭环（Round Loop）
每轮严格执行：
1. Plan：基于设备画像、已采集证据、历史轮次、经验库生成命令计划。
2. Approve：批准计划（可手动/自动）。
3. Execute：执行只读命令；同会话同设备同命令优先复用历史输出。
4. Analyze：把命令回显+结构化证据+历史信号交给 AI 判断。
5. StopDecision：判断收敛或进入下一轮。

## 5. 经验与历史如何指导下一步
- 计划阶段注入：
  - 历史轮次摘要（最近多轮）
  - 已执行命令缓存（避免重复）
  - 已知问题库 / 案例库 / SOP / 命令经验库（按开关）
- 分析阶段产出：
  - 证据链
  - 根因假设评分
  - StopDecision（是否建议收敛）
  - 下一轮建议命令（只读）

## 6. UI 交互原则（与流程对齐）
- AI 对话区只保留当前有效动作卡（不持久化旧按钮，避免刷新后错位）。
- 操作按钮按需弹出，动作来自后端 `next_action`。
- 自动批准开启后，匹配步骤自动推进；未匹配步骤明确提示“策略暂停点”。
- 右侧只看实时执行观测与设备回显，不再分散多处控制。

## 7. 急刹车与恢复
- “停止”只在用户输入框右侧。
- 点击后立即中断当前请求并进入急刹状态。
- 通过对话动作“继续”恢复；或“新会话”清空上下文重新开始。

## 8. 判定质量底线
- 不允许“证据不足却强结论”。
- 每轮必须给出证据链与下一步建议。
- 输出必须说明时间窗与设备时间校准关系。

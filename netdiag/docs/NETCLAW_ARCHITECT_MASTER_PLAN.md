# NetClaw 总体设计规划（架构师版）

## 1. 目标
把 NetClaw 设计为“网络工程师代理平台”，而不是“网络诊断表单工具”。

系统目标：
- 用户通过自然语言描述问题
- AI 主动理解、追问、执行、验证、收敛
- 诊断过程始终围绕设备证据展开
- 输出可解释、可回放、可复用

## 2. 系统角色
### 2.1 用户
- 提供现象、业务影响、已知时间线
- 可修正方向、补充信息、停止流程

### 2.2 AI 工程师
- 理解问题
- 制定并执行只读排障动作
- 基于证据收敛根因
- 给出下一步和最终结论

### 2.3 架构师
- 定义诊断内核、交互逻辑、验收门禁
- 审查实现是否偏离目标
- 仅以证据验收，不以“代码存在”验收

## 3. 核心架构
### 3.1 三层结构
1. `Agent Interaction Layer`
   - 对话理解
   - 动作卡生成
   - 用户追问与反馈
2. `Diagnosis Kernel Layer`
   - 假设生成
   - 命令意图选择
   - 证据抽取
   - 收敛判断
3. `Execution & Knowledge Layer`
   - 设备连接
   - 只读命令执行
   - 学习库 / 案例库 / 已知问题库 / Zabbix / 配置历史

### 3.2 核心闭环
`对话输入 -> 上下文建模 -> 假设 -> 最小命令 -> 执行 -> 证据 -> 结论/下一轮`

## 4. 产品形态
### 4.1 左侧对话区
- 用户和 AI 的主要交互界面
- 动作按钮只在对话中按需出现
- 每轮必须输出阶段结论

### 4.2 右上分析区
- 展示推理详情、假设、证据、结论
- 长文本折叠
- 点击聊天中的详情可切换到这里展示

### 4.3 右下终端区
- 左侧命令列表
- 右侧命令回显
- 当前执行项固定显示
- 历史命令可回看

## 5. 设计原则
1. AI 主导，不让用户配合字段表单。
2. 证据优先，不让大模型直接啃大段原始输出。
3. 最小命令集优先，不一次性执行大量命令。
4. 同会话命令去重复用。
5. 已有直接证据时立即收敛。
6. 模型失败时快速回退，不允许长时间卡住。

## 6. 分阶段规划
### Phase 1：Agent 启动能力
- 对话上下文合并
- 时间窗默认启动
- 工程师式追问
- 后端允许无明确时间窗启动会话

### Phase 2：假设驱动内核
- 从“大计划”改为“最小验证循环”
- 每轮只执行最有信息增益的命令
- 强制输出本轮结论

### Phase 3：证据工作台
- 分析区与终端区彻底分离
- 证据对象可点击展开
- 命令结果持久回放

### Phase 4：自适应代理
- 多设备跳转
- 动态时间窗
- 更强的经验库 / SOP / 历史状态注入

## 7. 架构师审查重点
1. 是否仍在要求用户填字段
2. 是否仍在一次性计划过多命令
3. 是否仍在重复执行同命令
4. 是否已有直接证据仍不收敛
5. 是否仅输出状态而不输出结论
6. 是否 UI 动作分散、不以对话为中心

## 8. 成功标准
- 用户一句话即可启动诊断
- 常见问题可在 30 秒内给出阶段结论
- 复杂问题在 2 分钟内给出有证据的本轮结论
- 会话过程可回放、可审计、可复用

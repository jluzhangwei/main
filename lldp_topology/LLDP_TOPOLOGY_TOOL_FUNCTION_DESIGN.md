# LLDP 拓扑工具功能设计与实现说明

## 文档编写计划

本文档不按页面组件逐项解释，而按用户实际工作链路拆解工具能力，目标是能同时用于技术说明、项目复盘和向团队解释设计细节。

1. 先说明工具解决的问题：多来源网络邻接数据采集、自动拓扑生成、链路性能分析和可视化运维。
2. 再说明数据如何进入系统：CLI、SQL、NDMP、CSV 粘贴/文件、Json 会话导入。
3. 然后说明拓扑如何生成：字段识别、设备身份归一、链路去重、坐标保留、布局与过滤。
4. 再展开拓扑编辑能力：选择、删除、恢复、撤销、手工节点、手工链路、链路描述。
5. 说明节点组能力：合组、颜色、组视图、组内拓扑坐标隔离、移出组。
6. 说明链路分析能力：CLI/Zabbix 利用率采集、带宽判断、颜色分析、负载分担分析、标签显示。
7. 最后说明导出、状态保存、性能优化和可靠性设计。

## 一、项目定位

LLDP 拓扑工具的核心目标是把网络设备的邻接关系、端口信息、链路带宽、链路利用率和人工维护信息整合成一张可操作的拓扑图。它不是单纯的画图工具，而是一个面向网络运维场景的数据采集、拓扑计算、链路分析和状态管理工具。

传统方式通常需要工程师手工登录设备、复制 LLDP 输出、整理 Excel、再用画图工具手工绘制拓扑。这个工具把这条链路自动化：

- 从设备、数据库、NDMP API、CSV、Json 会话中采集数据。
- 自动识别设备、端口、邻居和 IP。
- 自动生成拓扑节点和链路。
- 自动保留用户调整过的位置。
- 支持按业务、层级、名称相似度组织拓扑。
- 支持链路利用率和带宽展示。
- 支持节点组、路径查询、链路描述、手工补链路等运维编辑能力。
- 支持导出 PNG、draw.io、Mermaid、链路汇总 CSV、Json 会话。

## 二、整体数据模型

工具内部将所有导入方式统一成同一类 LLDP 行数据。标准字段包括：

- `depth`：递归层级。
- `localhostname`：本端设备名称。
- `ipaddr` / `源设备IP` / `sourceip`：本端设备 IP。
- `localinterface`：本端接口。
- `remotehostname`：邻居设备名称。
- `remoteinterface`：邻居接口。
- `目的设备Vendor` / `remotevendor`：邻居厂商。
- `目的设备IP` / `remoteip`：邻居 IP。

统一成标准字段后，前端建图逻辑不需要关心数据来自 CLI、SQL、NDMP 还是 CSV。所有来源最终进入同一条管线：

1. 解析输入。
2. 推断字段。
3. 合并或覆盖已有行。
4. 去重。
5. 生成节点和链路。
6. 保留已有坐标和视图状态。
7. 刷新拓扑。

这种设计的价值是：新增数据源时，只要后端或前端把结果转换成标准 CSV 字段，就可以复用现有建图、过滤、分组、链路分析、导出能力。

## 三、导入功能

### 3.1 CLI 导入

CLI 导入用于从真实网络设备实时采集 LLDP 邻接。它适合没有完整数据库、需要现场确认、或者希望获取最新邻接关系的场景。

#### 接入方式

CLI 导入支持多种接入路径：

- 直连 SSH：直接登录目标设备。
- SMC 跳板：先连接 SMC shell，再从 SMC 内部 SSH 到目标设备。
- PAM/跳板登录链路：通过命令模板和账号密码完成二段式登录。

页面上可以配置：

- 起始设备地址。
- 递归次数。
- 并发查询数量。
- 是否只递归 `172.*` 网段。
- 是否启用 ping 预检查。
- SSH 用户名和密码。
- SMC 地址、端口、命令模板。
- 登录超时时间。
- 命令执行超时时间。

#### ping 预检查

CLI 查询可以启用 ping 预检查。逻辑是：

1. 对目标设备 ping 2 个包。
2. 任意一个包可达，即认为目标可尝试登录。
3. 如果完全不可达，则跳过该设备，避免 SSH 登录长时间等待。

这个选项用于大规模递归查询时减少无效等待。它对不可达设备较多的场景非常有效，但如果网络环境禁止 ICMP，可以关闭该选项。

#### 登录与超时控制

登录阶段控制点包括：

- SSH `ConnectTimeout`。
- 登录交互超时。
- 命令执行超时。
- 不做多次重试，避免单台异常设备拖慢整体递归。

工具此前遇到过设备登录慢、空回显、命令卡住的问题，因此当前逻辑更偏向“快速失败”。也就是说，一台设备不可用，不应该阻塞整批拓扑采集。

#### 分页关闭

设备登录成功后，会先尝试关闭分页，再进行厂商探测和 LLDP 查询。

原因是部分 Cisco/NX-OS 设备执行 `show version` 时会出现 `--More--`，如果分页没有提前关闭，采集线程会持续等待输出，造成几十秒甚至分钟级延迟。

工具会尝试执行：

- Huawei：`screen-length 0 temporary`
- Cisco/NX-OS/Arista：`terminal length 0`

在识别厂商后，还会按厂商再做一次分页关闭，保证后续 LLDP 命令尽量一次性返回完整输出。

#### 厂商识别

CLI 导入会自动识别设备厂商，主要通过版本命令判断：

- Huawei：`display version` / `dis version`
- Cisco/NX-OS：`show version`
- Arista：`show version`

识别结果会影响后续 LLDP 命令选择、接口输出解析、设备名探测和利用率查询方式。

#### 设备名探测

设备名探测按优先级执行：

1. 从命令行 prompt 直接提取设备名。
2. Cisco 优先使用 `show hostname`。
3. 再回退到 `show version` / `show ver`。
4. Huawei 使用 `display current-configuration | include  sysname` 或 `dis current-configuration | include  sysname`。

这个顺序是为了避免在部分 Cisco 设备上执行 `show running-config | include ^hostname` 造成长时间空回显。优先从 prompt 和轻量命令获取名称，可以显著降低单设备探测耗时。

#### LLDP 命令与解析

CLI 查询会根据厂商执行不同 LLDP 命令，并由不同解析器解析：

- Huawei：支持 brief 表格和 detailed 输出。
- Cisco/NX-OS/IOS-XR：支持 `show lldp neighbors` 和 detail 类输出。
- Arista EOS：支持 Arista 格式的 LLDP detail 输出。

解析器会提取：

- 本端设备名。
- 本端接口。
- 邻居设备名。
- 邻居接口。
- 邻居 IP。
- 邻居厂商。
- 当前递归层级。

Huawei 设备中常见的 `100GE1/0/17:0` 这类子接口或 breakout 端口也会保留原始接口名称，避免后续链路判断时丢失端口维度。

#### 递归轮询

递归查询逻辑是按层级推进：

1. 第 0 层查询起始设备。
2. 从第 0 层结果里提取邻居 IP。
3. 第 1 层并发查询这些邻居。
4. 继续提取下一层邻居，直到达到用户设置的递归次数。

递归中会维护：

- 已查询设备集合。
- 当前层待查询设备。
- 下一层发现设备。
- 每台设备耗时。
- 每层摘要。
- 失败设备列表。

如果启用“只递归 172.*”，那么只有邻居 IP 以 `172.` 开头时才会进入下一层。这用于避免把管理地址、服务器地址、跨域地址全部卷入拓扑递归。

#### 并发控制

CLI 并发数量由用户设置，系统限制在合理范围内，避免一次性登录过多设备压垮跳板或网络设备。

状态栏会显示：

- 累计完成数量。
- 本层设备数量。
- 当前并发数量。
- 当前递归层。
- 已耗时。
- 当前正在处理的设备。

这个设计用于让用户判断慢是因为设备多、并发低、某台设备卡住，还是整层递归确实很大。

#### 去重逻辑

CLI 结果会做两类去重：

1. 行级去重：基于 `depth + 本端设备 + 本端接口 + 邻居设备 + 邻居接口`。
2. 设备归一：如果同一台设备既以 IP 出现，又以后续探测到的设备名出现，会尽量归一到同一个设备节点。

这样可以避免同一个真实设备在拓扑中出现“IP 节点”和“主机名节点”两份。

#### Debug 日志

CLI 查询会输出 debug 日志，便于定位慢点和解析问题。日志重点包括：

- 每层摘要：提交多少台、完成多少台、发现多少下一层、耗时多少。
- 每台设备耗时：设备、层级、厂商、邻居数、耗时、错误。
- 原始发送命令和回显片段。

当某台设备查询慢时，可以直接在 debug 里看它卡在登录、厂商探测、设备名探测、LLDP 命令还是解析阶段。

### 3.2 SQL 导入

SQL 导入用于从已有 LLDP 数据库中生成拓扑，适合有周期性采集平台或历史 LLDP 表的场景。

#### 查询方式

用户输入起始设备名，后端从数据库中查询该设备的最新 LLDP 邻接关系。

SQL 逻辑不是简单查一台设备，而是按递归层级做 BFS 查询：

1. 以起始设备为第一批查询对象。
2. 查询这些设备最新时间点的 LLDP 记录。
3. 取结果中的邻居设备作为下一层。
4. 重复直到达到最大递归层级。

#### 最新数据选择

SQL 查询会优先取每台设备最新 `create_time` 的记录，避免旧数据污染当前拓扑。

这样设计的原因是网络拓扑变化很频繁，如果混入历史 LLDP 数据，可能会把已经不存在的链路画出来。

#### 设备别名与 IP 归一

SQL 查询会根据最新 IP 映射处理别名。如果一个设备在数据库中存在多个名称或 IP 记录，查询逻辑会尝试把同一设备相关记录合并到同一个查询范围内。

这可以降低同一设备由于名称/IP不一致导致查不全的问题。

#### 取消能力

SQL 查询使用任务化接口：

- 创建任务。
- 前端轮询任务状态。
- 用户可以取消查询。

如果数据库查询异常慢，用户不需要重启服务，可以直接取消当前任务。

### 3.3 NDMP 导入

NDMP 导入用于调用接口获取设备 interface 信息，再从 interface 字段中提取 neighbor 相关数据。

当前接口形态是：

```http
POST /apis/ndmp/v2/device_interface/query
```

请求参数可以使用：

- `device_hostname`
- `device_ip`

#### 查询模式

页面支持：

- 自动识别。
- 按 hostname 查询。
- 按 IP 查询。

如果 hostname 查询失败，可以切换为 IP 查询；如果输入本身是 IP，也可以直接以 IP 查询。

#### 递归控制

NDMP 导入也支持递归次数，逻辑和 CLI 类似：

1. 查起始设备接口。
2. 从接口 neighbor 字段提取邻居设备名或 IP。
3. 继续查下一层。
4. 达到用户设置的轮询次数后停止。

NDMP 递归上限较低，避免 API 查询范围过大。

#### link_status 条件

NDMP 支持 `link_status == up` 作为可选过滤条件。

- 开启时：只看当前 up 的链路，更适合生成当前拓扑。
- 关闭时：历史 down 的链路也可能返回，更适合排查历史状态或对比曾经存在的连接。

#### 字段兼容

NDMP 返回结构可能不是固定字段名，因此后端会从多个候选字段中提取：

- 本端设备名。
- 本端设备 IP。
- 本端接口。
- 邻居设备名。
- 邻居设备 IP。
- 邻居接口。
- 邻居厂商。
- link status。

如果接口返回嵌套在 `data`、`result`、`payload`、`items`、`rows` 等字段里，解析逻辑也会尝试展开。

### 3.4 CSV 文件导入与粘贴导入

CSV 导入用于手工补充、第三方系统导出、Excel 整理后的数据。

支持两种方式：

- 选择 CSV 文件。
- 直接粘贴 CSV 文本。

#### 表头识别

导入时会自动识别表头，并根据字段名推断：

- 本端设备。
- 邻居设备。
- 本端接口。
- 邻居接口。
- 本端 IP。
- 邻居 IP。

支持逗号、Tab、分号作为分隔符，也支持带引号的字段。

#### 追加逻辑

CSV 可以作为新拓扑导入，也可以追加到当前拓扑。

追加时会：

1. 保留当前节点坐标。
2. 解析新 CSV。
3. 按端点和接口生成链路唯一键。
4. 已有链路则覆盖。
5. 新链路则新增。
6. 重新生成拓扑，但尽量保留已有可视位置。

这个逻辑用于手工补链路。例如发现两台设备之间缺一条链路时，可以通过 CSV 粘贴一行补进去。

### 3.5 Json 导入

Json 导入用于恢复工具导出的完整会话状态。

Json 不只是数据行，还包含：

- 原始 LLDP 行。
- 字段配置。
- 节点坐标。
- 删除状态。
- 节点组。
- 节点组内独立坐标。
- 手工节点。
- 手工链路。
- 链路描述。
- 自定义节点颜色。
- 当前 UI 设置。

Json 支持两种使用方式：

- 导入：替换当前工作集。
- 追加：把 Json 中的数据追加到当前工作集。

Json 追加时，已存在节点尽量使用当前视图中的位置，新节点会按导入内容或当前布局生成，并且新增节点会被选中，方便用户拖动整理。

## 四、拓扑生成

### 4.1 建图输入

拓扑生成的输入包括：

- 标准 LLDP 行。
- 手工新增节点。
- 手工新增链路。
- 节点组信息。
- 删除状态。
- 过滤状态。
- 利用率缓存。
- 自定义颜色。
- 链路描述。

建图时不会只看原始 CSV 行，而是把这些状态合并后生成当前可见图。

### 4.2 设备身份归一

设备可能以多种形式出现：

- 主机名。
- IP 地址。
- 带域名的 FQDN。
- 同一设备多个管理地址。
- 邻居侧只返回 IP，后续查询才返回名称。

工具会做以下处理：

- 去除域名后缀。
- 规范化大小写和空白字符。
- 优先显示更可读的主机名。
- 在 IP 和主机名指向同一设备时合并为一个节点。
- 同一设备多个地址时，在节点地址栏展示多个地址，而不是拆成多个节点。

这对 Arista、Cisco、Huawei 混合网络很重要，因为不同设备返回 neighbor 字段的格式并不统一。

### 4.3 链路身份与去重

链路的唯一性主要由以下字段决定：

- 本端设备。
- 本端接口。
- 邻居设备。
- 邻居接口。

链路去重时会尽量识别 A->B 和 B->A 是同一条物理链路的两个视角，避免同一条线重复显示。

如果两台设备之间存在多条不同接口链路，则它们是多条独立链路，不会错误合并成一条物理链路。

### 4.4 合并多条链路

两台设备之间有多条链路时，工具支持两种展示方式：

- 合并多条链路：显示为一条汇聚链路，标签里汇总端口、带宽和利用率。
- 不合并多条链路：每条物理链路独立显示。

合并状态不会改变原始数据。原始单条链路仍然保存在 `baseEdges` 中，只是页面展示时聚合为一条可视边。

### 4.5 坐标保留

建图时会优先保留已有坐标：

- 已存在节点保留用户拖动后的位置。
- Json 导入时恢复 Json 内保存的位置。
- 追加导入时，重复节点使用当前视图位置。
- 新节点再由布局算法或导入坐标决定位置。

这个设计是为了解决“大图反复重建后位置漂移”的问题。当前实现不是每次都销毁整图，而是尽量增量更新 Cytoscape 元素。

### 4.6 拓扑生成按钮

“生成拓扑”会基于当前工作集和当前布局配置重新计算拓扑。

如果在主视图中点击，它作用于整张拓扑。

如果在节点组视图中点击，它只作用于当前节点组视图内的节点和相关邻接设备，不应该影响主视图的整体坐标。

## 五、布局与分层

### 5.1 力导向布局

力导向布局适合未知结构或数据初次导入。它通过节点排斥、边长度、组件间距等参数让拓扑自动展开。

页面上的“分散强度”会影响：

- 节点间排斥力。
- 边长度。
- 组件间距。
- 布局边距。
- 重叠规避强度。

数值越大，拓扑越分散；数值过大时图会拉得很开，数值过小时链路和标签更容易重叠。

### 5.2 分层布局

分层布局通过关键字把设备放到不同层。例如：

- Core
- SuperSpine
- Spine
- Border
- Leaf
- ToR

用户可以用“分层关键字”定义层级。表达式规则是：

- 逗号：分层。
- 斜杠或竖线：同层多个关键字，满足任意一个即可。
- `&` 或 `+`：多个关键字必须全部命中。

例如：

```text
CORE,SUPERSPINE,SPINE,BORDER,LEAF/TOR
```

表示 Core 在上层，SuperSpine 在下一层，Leaf 和 ToR 可以在同一层。

### 5.3 分区布局

分区关键字用于把同一层中的节点再按业务或区域分区。例如：

- SG
- WAN
- SHOPEE
- GARENA
- DCI

分区逻辑与分层关键字一致，也支持逗号、斜杠和 `&`。

匹配顺序是从左到右，先匹配到的分区优先。这样用户可以把重要业务域放在更靠前的位置。

### 5.4 名称相似聚拢

名称相似聚拢用于让命名相近的设备靠近。

相似度计算不是简单字符串排序，而是综合考虑：

- 从左到右的公共前缀。
- 按 `-` 拆分后的 token 相似度。
- LAN/WAN/OOB/MGMT 这类平面关键字。
- R001/R003 这类编号差异。
- 角色关键字，如 Spine、Leaf、Core。

例如：

```text
TH-THSTT1-Garena-LAN-R001-Spine-01
TH-THSTT1-Garena-LAN-R003-Spine-02
```

会比：

```text
TH-THSTT1-Garena-LAN-R001-Spine-01
TH-THSTT1-Garena-WAN-R001-Spine-01
```

更接近，因为前者从左到右的业务平面、角色和命名结构更一致。

### 5.5 不显示与只显示

“不显示”和“只显示”是视图过滤，不会删除原始数据。

匹配逻辑使用统一的文本规范化：

- 忽略大小写。
- 兼容中文/英文输入。
- 兼容全角/半角字符。
- 兼容不同横线字符。
- 会同时匹配原始文本和去掉标点后的文本。

因此输入 `OOB` 应该能匹配 `-OOB`，避免因为短横线导致过滤失败。

过滤对象包括：

- 节点完整名称。
- 节点短名称。
- 节点 IP。
- 节点组成员名称。
- 节点组标签。

### 5.6 172 网段显示过滤

页面支持只显示 `172.*` 相关节点或只递归查询 `172.*`。

两者含义不同：

- 只显示 `172.*`：只是前端视图过滤，不改变数据。
- 只递归 `172.*`：采集时控制下一层递归范围，会影响实际采集到的数据。

## 六、节点操作

### 6.1 设备选择

用户可以直接点击节点选择设备，也可以框选多个节点。选择结果会影响：

- 删除。
- 保留选中。
- 相邻保留。
- 设备合组。
- 对齐操作。
- 手工增加链路。
- 自定义颜色。
- 路径查询默认源/目的。

### 6.2 定位设备

定位设备用于在大图中快速找到目标。它与过滤使用同一套匹配表达式：

- 逗号分组。
- 斜杠表示同层任意命中。
- `&` 表示必须全部命中。
- 中英文输入规范化。
- 标点差异兼容。

定位不会隐藏其他节点，只会高亮命中的设备并移动视图。

### 6.3 删除与恢复

删除设备不会清除原始数据，而是把节点加入删除栈。

恢复删除设备时，会从删除栈中恢复最近删除的节点。这样用户可以临时清理视图，但不破坏导入数据。

### 6.4 撤销与恢复

页面支持最多 5 步撤销/恢复。

快照内容包括：

- 原始数据行。
- 当前坐标。
- 删除栈。
- 过滤状态。
- 节点组。
- 手工节点。
- 手工链路。
- 链路描述。
- 自定义颜色。
- 选择状态。

对齐、移动、删除、合组等操作都应该进入撤销历史。

### 6.5 单链路节点

“单链路节点”用于找出当前视图中只有一个邻接设备的节点。

逻辑是：

1. 只看当前可见节点和可见边。
2. 计算每个节点的唯一邻接设备数量。
3. 如果一个节点只有一个邻接设备，则标记为单链路节点。
4. 如果节点组内包含单链路设备，组节点也会变色提示。

该功能不会选择链路，只选择和高亮符合条件的节点。

### 6.6 对齐与排布

对齐功能用于手工整理局部拓扑。支持：

- 水平对齐。
- 垂直对齐。
- 左对齐。
- 右对齐。
- 水平等距。
- 垂直等距。

这些操作只作用于当前选中节点，并写入坐标状态，支持撤销。

## 七、手工补充能力

### 7.1 添加普通节点

右键菜单支持添加普通节点。普通节点用于补充概念设备、未采集设备或临时标注对象。

新增节点会进入 `manualCanvasNodes`，参与后续：

- 拖动。
- 选择。
- 手工加链路。
- Json 导出。
- Json 导入恢复。

### 7.2 添加 Cloud 节点

Cloud 节点用于表达外部网络、Internet、第三方网络、运营商云、未知域等概念。

Cloud 节点和普通设备不同：

- 使用云图标。
- 可修改名称。
- 用于概念图表达。
- 不作为真实 LLDP 设备参与递归。

Cloud 节点同样会进入 Json 会话，刷新或导入后应该恢复。

### 7.3 手工增加链路

当用户选中两个节点后，右键可以增加链路。

弹窗需要输入：

- 源设备接口。
- 目的设备接口。

保存后，工具会生成一行标准 LLDP 数据，并追加到当前工作集。这样手工链路和采集链路使用同一套建图逻辑，不会成为孤立的前端图形。

### 7.4 链路描述

右键单条链路可以添加描述。

描述会作为链路数据的一部分保存，用户可以通过“显示链路描述”复选框控制是否显示。

如果是合并链路，描述可以绑定到设备对；如果是单条链路，描述绑定到具体端口链路。

描述标签会与端口、带宽、利用率一起参与链路标签展示，避免独立浮动导致信息难以关联。

## 八、节点组功能

### 8.1 设备合组

节点组用于把多个设备折叠成一个逻辑节点，适合大规模拓扑中的区域、机房、Pod、业务组或重复结构。

支持：

- 多选普通节点合组。
- 多选节点组和普通节点再次合组。
- 单个节点组保持原名称。
- 多个节点组重新计算组名。

合组后，组节点使用黄色，避免与普通设备颜色、单链路红色冲突。

### 8.2 组名计算

当多个设备合组时，工具会从设备名称中提取公共部分生成组名。

例如多个设备都包含：

```text
TH-THSTT1-Garena-LAN
```

则组名会尽量保留这些公共业务信息，而不是生成无意义的随机名称。

用户也可以右键修改组名。

### 8.3 组内设备数量

组节点名称下方可以显示组内设备数量。

这用于大图中快速判断一个组代表多少真实设备，避免折叠后信息完全丢失。

### 8.4 合组后的链路处理

节点合组后，组内成员的外部链路会映射到组节点。

如果多个成员分别连接到同一外部设备，组节点到外部设备之间可能显示为合并链路。原始成员链路不会丢失，只是视图上折叠。

### 8.5 进入节点组视图

双击节点组可以进入节点组视图。

组视图包含：

- 组内所有设备。
- 组内设备的一跳邻接设备。
- 与组内设备相关的链路。

这样用户可以在不展开主图所有内容的情况下，专注查看一个节点组内部结构。

### 8.6 组视图坐标隔离

主视图和节点组视图使用独立坐标。

原因是同一批设备在主视图中需要服务整体拓扑，在组视图中需要服务局部分析。如果两者共用同一套坐标，用户在组视图整理位置会破坏主视图布局。

当前设计是：

- 主视图保存一套坐标。
- 每个节点组视图保存单独的 scope 坐标。
- 进入组视图时读取该组自己的坐标。
- 退出组视图时保存组内坐标，但不覆盖主视图坐标。
- Json 导出时同时保存主视图和组视图坐标。

旧 Json 如果没有组视图坐标，也可以导入。系统会使用主视图坐标初始化组视图，不会报错。

### 8.7 组内右键操作

节点组视图内的设备右键支持：

- 返回主视图。
- 移出节点组。
- 复制名称/地址。
- 当前利用率。
- 区间利用率。
- 自定义颜色。

移出节点组时会修改节点组成员关系。退出主视图后，该设备不应该再回到原节点组。

## 九、节点颜色

### 9.1 默认角色颜色

工具会根据设备名称和角色给节点设置基础颜色，例如：

- Core。
- SuperSpine。
- Spine。
- Border。
- Leaf。
- ToR。
- SBB。

这让用户在不查看完整名称的情况下，也能快速识别设备角色。

### 9.2 关键字颜色规则

节点颜色可以按名称关键字自动定义，例如：

```text
SUPERSPINE:#4f46e5
SPINE:#2563eb
BORDER:#06b6d4
LEAF:#16a34a
TOR:#22c55e
CORE:#7c3aed
SBB:#64748b
```

规则按关键字命中设备名称后应用颜色。

### 9.3 右键自定义颜色

用户可以选择一个或多个节点，右键设置自定义颜色。

自定义颜色优先级高于默认角色颜色，但不覆盖特殊状态色：

- 节点组黄色。
- 单链路节点红色。
- 选中状态边框。

这样可以保证功能提示色不会被业务颜色规则误覆盖。

## 十、路径查询

### 10.1 查询目标

路径查询用于在当前可见拓扑中寻找两台设备之间的路径。

它不依赖后端，不重新采集数据，只基于当前 Cytoscape 图中的可见节点和可见边计算。

### 10.2 查询条件

弹窗支持：

- 源节点。
- 目的节点。
- 必经点。
- 备份路径数量。

如果当前选中两个节点，弹窗会自动填入源和目的。选中三个及以上节点时，前两个作为源和目的，其余作为必经点。

### 10.3 搜索规则

路径按无向图处理，按跳数最少计算。

必经点按输入顺序串联：

```text
源 -> 必经点1 -> 必经点2 -> 目的
```

任何一段不可达时，会明确提示失败段。

### 10.4 等长主路径

主路径不只支持一条。如果存在两条跳数相同的最短路径，结果会显示：

- 主路径。
- 主路径 2。

这样可以发现等价路径，而不是只看到算法第一次找到的那一条。

### 10.5 备份路径

备份路径支持 0、1、2 条。

备份路径会优先避开已经使用过的边，也就是边级去重优先。但它不是严格的 disjoint path 算法，不保证所有节点和链路完全独立。

### 10.6 只保留路径节点

路径查询结果中可以点击“只保留路径节点”。

逻辑是：

1. 找出所有路径中的节点。
2. 删除当前可见图中不属于路径的节点。
3. 删除动作进入删除栈。
4. 用户可以通过恢复删除设备找回。

这个功能适合从大图中裁剪出一段路径用于分析或导出。

### 10.7 右键设为必经点

在路径结果存在时，用户可以右键节点设为必经点，然后重新计算路径。

这使路径查询可以从“先找一条路”逐步变成“指定必须经过某些网络节点的路径”。

## 十一、链路利用率分析

### 11.1 分析目标

链路利用率分析用于把拓扑从“连接关系图”升级为“运行状态图”。

工具可以展示：

- TX 利用率。
- RX 利用率。
- 时间段最大值。
- 时间段最小值。
- 总带宽。
- 负载分担差异。
- 拥塞颜色。

### 11.2 CLI 利用率采集

CLI 利用率查询会登录设备并查询接口状态。

不同厂商命令不同：

- Huawei：查询接口输出中的 input/output rate 和 bandwidth。
- Cisco/NX-OS：查询 interface 详情中的 input/output rate、load、bandwidth。
- Arista：查询 show interface 输出中的速率和带宽。

解析后会得到：

- TX 使用百分比。
- RX 使用百分比。
- TX bps。
- RX bps。
- 接口带宽。

如果设备输出里没有直接给百分比，但有 bps 和 bandwidth，工具会自行计算：

```text
利用率 = 当前 bps / 接口带宽 * 100
```

### 11.3 Zabbix 利用率采集

Zabbix 查询用于从监控系统获取链路历史或当前利用率。

配置包括：

- Zabbix URL。
- Token。
- SSL 校验。

配置会保存到服务器本地，用户不需要每次重新输入。

### 11.4 Zabbix 主机匹配

Zabbix 查询会尝试用多种方式匹配主机：

1. 通过设备 IP 匹配 host interface。
2. 通过设备名称匹配 host/name。
3. 如果一个设备存在多个候选 host，按数据完整度选择最优项。

最优项判断会考虑：

- 是否有 speed item。
- 是否有 TX item。
- 是否有 RX item。
- 是否与输入 IP 更匹配。
- item 数量是否更完整。

### 11.5 接口名称兼容

不同系统中接口名称可能不同，工具会做别名展开，例如：

- `Eth1/1` 与 `Ethernet1/1`
- `Gi` 与 `GigabitEthernet`
- `Te` 与 `TenGigE`
- `Hu` 与 `HundredGigE`
- `BE` 与 `Bundle-Ether`
- `Po` 与 `Port-Channel`
- `Lo` 与 `Loopback`
- `MEth` 与 `MgmtEth`

这个设计用于解决“拓扑里有端口，但 Zabbix 查不到 item”的问题。

### 11.6 时间窗查询

Zabbix 支持：

- 当前值。
- 时间段最大值。
- 时间段最小值。

时间段查询会根据时间跨度选择：

- 较长时间：优先使用 trend 数据。
- 较短时间或 trend 无数据：回退到 history 数据。

这可以适配 Zabbix 对历史数据压缩的情况。如果最近没有 history，但三个月前 trend 有数据，也能尽量查到。

### 11.7 带宽判断

链路带宽来源按优先级判断：

1. 已有缓存中的接口带宽。
2. Zabbix speed item。
3. 更长时间窗口中的 speed 数据。
4. 对端接口 speed。
5. 从接口名称推断，例如 `100GE`、`400GE`、`TenGigE`。

这样可以减少每次查询都重复查询带宽，也可以在某一侧 speed 缺失时利用对端信息补齐。

### 11.8 利用率口径

查询口径包括：

- TX。
- RX。
- MIN。
- MAX。

其中：

- `时间段最大值` 表示在时间窗内取最大使用率。
- `时间段最小值` 表示在时间窗内取最小使用率。

MIN 和 MAX 不应该混淆。查询时会按不同聚合方式分别从 Zabbix trend/history 中取值。

### 11.9 TX/RX 方向

TX 按链路箭头方向解释。

如果链路方向是：

```text
A -> B
```

那么标签中的 TX 表示 A 端口发往 B 方向的流量口径，RX 表示 A 端口接收方向。

页面支持同时显示 TX/RX，便于分析双向链路。

### 11.10 总带宽显示

总带宽显示不依赖“显示利用率标签”。用户可以只打开“显示总带宽”，在不显示利用率百分比的情况下查看链路容量。

如果多条链路合并显示，总带宽是多条链路带宽之和。

如果关闭合并多条链路，则每条物理链路展示自己的带宽。

### 11.11 链路拥塞分析

“链路拥塞分析”根据链路利用率给边着色：

- 绿色：低利用率。
- 黄色：中等利用率。
- 红色：高利用率。
- 灰色：无数据。

当前阈值逻辑为：

- 绿色 `< 40%`
- 黄色 `40% - 70%`
- 红色 `>= 70%`

这是从拥塞风险角度设计的，适合发现高负载链路。

### 11.12 负载分担分析

“负载分担分析”用于检查两台设备之间多条链路是否均衡。

逻辑是：

1. 找到同一设备对之间的多条链路。
2. 取每条链路当前口径下的利用率。
3. 计算最大值和最小值差值。
4. 根据差值着色。

阈值为：

- 绿色 `< 5%`
- 黄色 `10% - 20%`
- 红色 `>= 20%`

`5% - 10%` 属于观察区，可以不强调为异常。

链路拥塞分析和负载分担分析是二选一的，因为它们都是通过链路颜色表达不同含义。如果同时开启，会造成颜色语义冲突。

### 11.13 利用率过滤

利用率过滤可以只显示超过某个阈值的链路。

过滤逻辑会隐藏低于阈值的链路，并隐藏没有可见链路的孤立节点。它是视图过滤，不删除原始数据。

### 11.14 缓存

利用率查询结果会缓存，避免重复查询同一设备同一端口。

缓存价值包括：

- 减少 Zabbix API 压力。
- 减少 CLI 登录次数。
- 页面刷新后能恢复之前看到的利用率。
- 带宽存在时优先复用，不必每次重新查询。

## 十二、右键菜单设计

右键菜单按功能分组，避免所有操作堆在一起。

分组包括：

- 编辑类：固定位置、删除设备、删除链路。
- 路径/分组类：设为必经点、设备合组、展开合组、修改组名、返回主视图、移出节点组。
- 颜色类：自定义节点颜色。
- 复制类：复制名称、复制地址、复制链路。
- 查询类：当前利用率、区间利用率、追加查询。
- 创建类：添加节点、添加 Cloud、添加链路、添加链路描述。

菜单位置会根据鼠标位置和页面边界自动调整。如果在页面底部右键，菜单会向上展开，避免超出浏览器窗口。

## 十三、导出能力

### 13.1 PNG 导出

PNG 导出用于快速截图分享当前拓扑视图。

导出内容以当前可见画布为基础，适合即时沟通和汇报。

### 13.2 draw.io 导出

draw.io 导出用于把当前拓扑转为可编辑图形。

导出文件保留：

- 当前可见节点。
- 当前可见链路。
- 节点位置。
- 链路标签。
- 曲线边。

用户可以在 draw.io 中打开后继续编辑。

### 13.3 Mermaid 导出

Mermaid 用于快速生成文本化拓扑图。

导出内容基于当前显示结果，不单独定制另一个拓扑视图。当前页面上显示什么，Mermaid 中就尽量表达什么。导出结果为 `.mmd` 文件，便于在 draw.io、Markdown 或 Mermaid 渲染环境中继续使用。

### 13.4 链路汇总 CSV

链路汇总导出用于把拓扑中的链路信息给到审计、复盘或容量分析。

如果合并多条链路开启，导出字段包括：

- 源节点。
- 目的节点。
- TX。
- RX。
- 带宽。

如果合并多条链路关闭，导出字段包括：

- 源节点 + 源端口。
- 目的节点 + 目的端口。
- TX。
- RX。
- 带宽。

导出文件还会写入查询上下文：

- 数据来源。
- 当前口径。
- 查询模式。
- 时间范围。
- 导出时间。

如果数据来自 CLI，会标明 `data_source=CLI`；如果来自 Zabbix，会标明 Zabbix 查询条件。

### 13.5 Json 导出

Json 导出用于完整保存当前工作状态。

它不是普通 CSV，而是一个可恢复会话，包含：

- 数据行。
- 坐标。
- 组信息。
- 组视图坐标。
- 删除状态。
- 自定义颜色。
- 链路描述。
- 手工节点和链路。
- UI 设置。

适合跨浏览器、跨机器或后续继续编辑。

## 十四、状态保存与刷新恢复

页面会保存当前会话状态，避免刷新后内容丢失。

刷新恢复包括：

- 当前导入数据。
- 节点坐标。
- 删除和过滤状态。
- 节点组。
- 手工节点。
- 链路描述。
- 利用率显示。

但用户显式导入新 Json 时，会清理旧页面会话，避免刷新后又恢复到更早的状态。

## 十五、性能设计

### 15.1 增量更新 Cytoscape

大图性能的关键是避免每次操作都销毁整张图。

当前实现会尽量：

- 复用已有 Cytoscape 实例。
- 只添加新增元素。
- 只删除缺失元素。
- 更新已有节点和边的数据。
- 保留选择状态。
- 保留 viewport。

这比 `cy.destroy()` 后整图重建稳定得多，尤其是几千条链路的大图。

### 15.2 大图渲染模式

当边数量较多时，工具会启用更轻的渲染模式：

- 降低像素比。
- 拖动时隐藏重边样式。
- 拖动时减少箭头和标签渲染。
- 大图下使用更轻量的边曲线。

目的是保证拖动和点选不会被大量边标签拖慢。

### 15.3 标签按需显示

端口、利用率、TX/RX、总带宽、链路描述都通过复选框控制。

大图中标签越多，渲染压力越大。用户可以按当前分析目标打开需要的信息，而不是所有信息同时显示。

### 15.4 采集侧提速

CLI 采集的性能优化包括：

- ping 预检查。
- 无重试快速失败。
- 分页关闭前置。
- 厂商探测短超时。
- 设备名探测短超时。
- 并发查询。
- 每层递归控制。
- debug 定位慢设备。

这些优化解决的是“采集慢”，不是“画图慢”。采集和渲染分别优化，才能从根本上提高体验。

## 十六、可靠性与可维护性

### 16.1 数据源隔离

CLI、SQL、NDMP、CSV、Json 的采集方式不同，但最终统一为标准 LLDP 行。这样可以让采集逻辑和前端拓扑逻辑解耦。

### 16.2 原始数据不轻易破坏

过滤、隐藏、路径裁剪、删除设备，大多数都不直接删除原始数据，而是维护视图状态或删除栈。

这保证用户在探索过程中可以恢复，降低误操作成本。

### 16.3 人工编辑可持久化

手工节点、Cloud、手工链路、链路描述、自定义颜色、节点组都会进入 Json 导出。

这让工具不只是“看一次拓扑”，而是可以长期维护一张有业务语义的网络图。

### 16.4 Debug 可追溯

CLI 和利用率查询都有 debug 输出。遇到数据缺失时，可以区分：

- 是登录失败。
- 是命令无输出。
- 是解析失败。
- 是设备没有 neighbor IP。
- 是 Zabbix host 匹配失败。
- 是 Zabbix item 缺失。
- 是 bandwidth 缺失。
- 是时间窗内没有数据。

这比只提示“无数据”更利于排障。

## 十七、核心价值总结

这个工具的价值不只是“自动画拓扑”，而是把网络拓扑维护中的多个痛点连成闭环：

1. 多来源采集：CLI、SQL、NDMP、CSV、Json 都能进入同一拓扑管线。
2. 自动递归：从一台设备出发，按 LLDP 邻接自动扩展多层网络。
3. 数据治理：设备名/IP 归一、链路去重、历史数据隔离、字段自动识别。
4. 可视布局：力导向、分层、分区、名称相似聚拢、手工对齐。
5. 运维编辑：删除恢复、撤销恢复、手工补节点、补链路、链路描述。
6. 规模化管理：节点组、组视图、组内坐标隔离、组内设备维护。
7. 性能分析：CLI/Zabbix 利用率、带宽识别、拥塞分析、负载分担分析。
8. 结果复用：PNG、draw.io、Mermaid、链路汇总 CSV、Json 会话。

最终效果是：工程师可以从“手工登录设备、复制命令、整理表格、手动画图”，转为“输入起点、自动采集、自动生成、按需分析、导出复用”。这能显著降低拓扑维护时间，也能减少因为历史数据、重复链路、设备别名和人工绘图造成的错误。
